## Phase 3 產品驗證方法論

執行 `/ship` Phase 3 時，你必須對 ECHO 的 PRD 進行完整審查。**不是橡皮圖章，是真正找問題。**

### 驗證框架：五個核心問題

```
1. 用戶旅程連貫性
   每個用戶故事是否能從開始走到結束？
   有沒有缺失的步驟讓用戶卡住？
   錯誤路徑有沒有出口？

2. 功能優先順序合理性
   P0 的每個功能是否真的是「沒有就無法使用」？
   有沒有被誤列為 P0 但其實是 P1 的功能？
   有沒有被遺漏的真正關鍵功能？

3. 驗收標準可測試性
   每個 AC（驗收標準）是否具體、可驗證？
   是否有「系統應該運作正常」這種無法驗證的 AC？
   邊界條件（空狀態、錯誤狀態）是否在 AC 中？

4. 用戶心智模型一致性
   功能的命名和用戶的語言是否一致？
   操作流程是否符合用戶的直覺預期？
   有沒有「只有工程師才會這樣想」的設計？

5. 範疇蔓延風險
   PRD 有沒有悄悄加入超出原始需求的功能？
   「順便做」的功能是否已被正確分類為 P1/P2？
```

## Jobs-to-be-Done 分析框架

每個新功能，先用 JTBD 確認它真正在解決什麼問題：

```
情境（When）：用戶在什麼時候、什麼情境下使用？
動機（Motivation）：用戶真正想達成的目標是什麼？
預期結果（Expected Outcome）：成功完成後的狀態？
競爭替代品（Competing Solutions）：用戶現在怎麼解決這個問題？
```

範例：
```
功能：「匯出 Excel」
JTBD 分析：
  When：每週一需要向主管報告數據
  Motivation：不是「想要一個 Excel 檔」，而是「想讓主管看懂數字」
  Expected Outcome：主管認可數據，會議順利
  Competing：現在手動複製貼上到 Excel，花 30 分鐘

洞察：如果我們提供「一鍵產出 PPT 格式報告」可能比 Excel 更符合 JTBD。
這個洞察應該在 Phase 3 提出，在 Phase 2 只是照樣實作「匯出 Excel」。
```

## 功能優先排序：RICE 框架

當 ECHO 的優先排序有爭議時，用 RICE 客觀評估：

```
R = Reach（影響用戶數）
I = Impact（對每個用戶的影響程度，1-3）
C = Confidence（對估算的信心程度，0-100%）
E = Effort（工程工作量，人週）

RICE Score = (R × I × C) / E

Score 最高的功能優先做。
```

- 確認 PRD 裡的所有頁面路由和 API 端點清單完整
- 這些是 NEXUS 設計架構的輸入，缺一不可

- 「如果只能做一件事，哪件事影響最大？」
- 「如果刪掉這個功能，有多少用戶會真的受影響？」
- 「三個月後，用戶還會記得這個功能嗎？」
