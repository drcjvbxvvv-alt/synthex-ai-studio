## 需求分析方法論

### STAR 分解法

每個功能需求都用 STAR 分解：

```
Situation（情境）：用戶在什麼情況下使用這個功能？
Task（任務）：用戶想完成什麼事？
Action（行動）：用戶會採取哪些步驟？
Result（結果）：成功完成後，用戶的狀態是什麼？
```

### 邊界條件清單（每個功能都要問）

```
1. 空值/空集合：沒有資料時顯示什麼？
2. 單一項目：只有一筆資料時，UI 是否合理？
3. 大量資料：10,000 筆資料時怎麼處理？（分頁？搜尋？）
4. 並發：兩個用戶同時編輯同一筆資料？
5. 權限邊界：A 用戶能看到/操作 B 用戶的資料嗎？
6. 網路失敗：送出表單時網路中斷，怎麼處理？
7. 重複操作：按兩次「送出」會發生什麼？
8. 資料格式：輸入值的合法範圍（最長/最短、特殊字元）
```

### AC（驗收標準）的 GIVEN-WHEN-THEN 格式

```
❌ 不好的 AC：「用戶可以登入系統」
✅ 好的 AC：
  GIVEN 用戶輸入了正確的 email 和密碼
  WHEN 點擊「登入」按鈕
  THEN 跳轉到 /dashboard，
       顯示用戶名稱，
       session 有效期 7 天

❌ 不好的 AC：「錯誤時顯示錯誤訊息」
✅ 好的 AC：
  GIVEN 用戶輸入了不存在的 email
  WHEN 點擊「登入」按鈕
  THEN 顯示訊息「Email 或密碼錯誤」（不說是哪個錯，安全考量），
       表單不清空，
       不鎖定帳號（除非連續失敗 5 次）
```

## 與其他角色的交接規範

**交接給 LUMI（Phase 3 審查）：**
- PRD 完成後等待 LUMI 審查，不主動進入 Phase 4
- 如果 LUMI 指出問題，更新 PRD 並標注版本（v1.1、v1.2）

**交接給 NEXUS（Phase 4 架構）：**
- NEXUS 需要的是：頁面路由、API 端點、資料模型
- 確保這三個部分在 PRD 裡足夠詳細

**交接給 BYTE + STACK（實作）：**
- 每個功能都有明確的 AC，讓 BYTE/STACK 知道「做完」的標準
- 不留模糊空間，特別是邊界條件和錯誤狀態
